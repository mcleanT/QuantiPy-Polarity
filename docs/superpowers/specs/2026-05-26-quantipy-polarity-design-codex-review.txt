**Date:** 2026-05-26

## 1. Purpose

Package the existing Python reimplementation of QuantifyPolarity (QP) — currently developed in `Hughes Lab/Sachin/Polarity Quantification/` — as a standalone public repository: **QuantiPy Polarity**.

The single-shot promise: a user feeds in raw microscopy images and receives per-FOV polarity maps, rose plots, quantitative tables, migration-front overlays, and a self-contained HTML report — without needing to install the original QP `.app` binary or understand the staged-pipeline internals.

## 2. Goals and non-goals

**Goals:**
- One CLI command (`quantipy run`) takes raw images to all outputs.
- Fully marker-agnostic — works for any planar polarity marker (Vangl, Frizzled, Celsr1, Crumbs, etc.) via config.
- Three input entry points: `.nd2`, multi-page `.tif`, or pre-segmented label masks.
- Python implementation only — no vendored QP binary.
- Claude-Code-native: `CLAUDE.md`, `.claude/settings.json`, and three slash commands ship in-repo.
- Ship validation evidence (QP-vs-Python linear-relationship figure) so reviewers can verify equivalence.
- MIT license, single-author authorship.

**Non-goals (v0.1.0):**
- PyPI publishing (deferred to v0.2).
- Vendor the QP `.app` binary (cite the original tool instead).
- Generalize to non-planar-polarity quantification (e.g., apical-basal).
- Support `.lif` / `.czi` / `.lsm` input formats (deferred; can add later via `aicsimageio`).
- mkdocs documentation site (plain markdown is sufficient for v0.1.0).

## 3. Scope decisions (resolved during brainstorming)

| Decision | Resolution |
|---|---|
| Scope | Core pipeline + curated analyses (~25 files) |
| Entry point | CLI-first (`quantipy ...`), Python API exists for free via clean package layout but is not heavily advertised |
| Input formats | `.nd2`, multi-page `.tif/.tiff`, pre-segmented label masks |
| Generality | Fully generic — channel indices and markers configurable via YAML |
| Outputs | Per-FOV polarity maps + rose plots + quantitative tables + migration overlays + HTML report |
| QP binary | Python-only; drop vendored `.app`; cite original QP paper |
| License | MIT |
| Sample data | Tiny synthetic fixture in-repo + real FOV via GitHub Releases (not git-lfs) |
| Claude-Code integration | `CLAUDE.md` + `.claude/settings.json` + three custom slash commands |
| Repo home | Personal account (`mcleanT/QuantiPy-Polarity`), single-author, with `CITATION.cff` |
| Packaging | Approach A — clean `src/` Python package layout, CLI documented as primary entry |
| Validation | Dedicated `validation/` subpackage shipping the QP-vs-Python linear-relationship figure as static evidence |

### Layout rationale

- `src/` layout (not flat) prevents accidental local-import wins during tests and is the standard for PyPI-publishable scientific packages.
- Subpackages organized by **concern** (`io/`, `segment/`, `polarity/`, ...) rather than stage number. Pipeline ordering is defined in `cli.py`, not in directory names. Python module names cannot begin with digits, so this is also a hard requirement for the Python API to exist at all.
- `analyses/` and `validation/` are peers to the core modules — `analyses/` is the "curated analyses" bucket from the scope decision; `validation/` is the dedicated home for QP-vs-Python evidence.
- `.claude/commands/` ships three slash commands. `.claude/settings.json` pre-approves read-only Bash so users avoid permission prompts on first run.

## 5. CLI surface

All commands installed via `pyproject.toml`'s `[project.scripts]` as `quantipy = "quantipy_polarity.cli:main"`.

| Command | Wraps module | Purpose |
|---|---|---|
| `quantipy run` | All stages | One-shot: raw images → all outputs |
| `quantipy ingest` | `io/` | nd2/tif → normalized per-FOV TIFs (auto-detects format) |
| `quantipy segment` | `segment/` | Cellpose-SAM → label masks |
| `quantipy polarity` | `polarity/` | Label masks + membrane channel → per-cell axes |
| `quantipy front` | `migration/` | Migration front + local arrows; `--auto` or `--interactive` |
| `quantipy aggregate` | `polarity/per_cell.py` | Per-FOV parquet → experiment parquet |
| `quantipy plot` | `viz/` | Rose plots, polarity maps, overlays |
| `quantipy report` | `report/` | Build self-contained HTML report |
| `quantipy debug` | `interactive/` | Interactive per-cell viewer |
| `quantipy analyze <name>` | `analyses/<name>` | Run a curated analysis from the registry |
| `quantipy validate` | `validation/` | Regenerate QP-vs-Python linear-relationship figure |
| `quantipy download-demo` | — | Pull demo FOV from latest GitHub Release |

Every subcommand accepts `--config <path>`, `--input <path>`, `--output <path>`. `quantipy run` is the documented single-shot path; the others exist for redoing one stage without rerunning the full pipeline.

### Input auto-detection

`quantipy run --input ./raw` inspects the input directory:
- `*.nd2` files present → `io/nd2` path (full pipeline including stage 1)
- `*.tif/*.tiff` files only → `io/tif` path (skip ND2 conversion; validate dimensions/channels)
- Label-mask TIFs paired with membrane TIFs → `io/masks` path (skip ingest + segmentation, jump to `polarity/`)

Explicit override available via `input.source` in config (`nd2 | tif | masks`) for ambiguous directories.

## 12. Sample data

### Synthetic (in-repo)

`tests/fixtures/synthetic_fov.tif` (~1–3 MB):
- Procedurally generated by committed `tests/fixtures/_build.py` (Voronoi cells with known per-cell polarity ground truth).
- Doubles as regression fixture: `test_polarity.py` asserts boundary-PCA recovers seeded axis within tolerance.
- CI runs full `quantipy run` end-to-end on this fixture in <30s with no binary download.

### Real demo FOV (GitHub Releases)

- One clean FOV from the optoCelsr dataset (likely C10/FOV_04) uploaded as a Release asset on the v0.1.0 tag.
- Bundle: `demo_C10_FOV04.zip` containing raw `.nd2` + reference outputs.
- `quantipy download-demo` fetches from `https://github.com/mcleanT/QuantiPy-Polarity/releases/latest/download/demo_C10_FOV04.zip`, caches to `~/.cache/quantipy/demo/`, returns the local path.

## 15. Carve-out plan from `Hughes Lab/Sachin/Polarity Quantification/`

The existing dir is research-active. Plan must not break it.

1. **New repo lives at `Science/QuantiPy-Polarity/`** — sibling to `Hughes Lab/`, independent git repo.
2. **Copy + refactor, do not move.** Source dir stays intact. If something breaks during refactor, the original still runs.
3. **Refactor in batches via subagents** (per global CLAUDE.md mandate). Each subpackage = one subagent task: `io/`, `segment/`, `polarity/`, `migration/`, `viz/`, `report/`, `interactive/`, `analyses/`, `validation/`, plus CLI batch and tests batch.
4. **Strip optoCelsr-specific naming** as files move. Concretely: hard-coded `C10`/`D11` clone identifiers become user-supplied condition labels (read from config or directory names); `FOV_NN` stays as generic numbering; `Celsr1`/`optoCelsr` references in docstrings and variable names become "polarity marker" / "membrane channel"; magic channel indices (e.g., `img[..., 1]`) read from `input.channel_membrane` / `input.channel_segmentation`. Search for and remove every occurrence of `C10`, `D11`, `optoCelsr`, `Celsr1`, `Sachin`, `hPGK`, `pGK` outside of `examples/configs/celsr_optoclones.yaml` (which intentionally documents the original analysis).
5. **Tests must pass on new repo before pushing to GitHub.** Cut v0.1.0 tag once green. Upload demo + validation parquet to the Release.
6. **Original Hughes-Lab dir untouched.** Optionally later: one-line `README.md` there pointing to QuantiPy-Polarity. Not part of this plan.

## Summary Verdict
**Revise-and-proceed.** The spec has the right center of gravity: CLI-first, reproducible outputs, real validation evidence, and a clean `src/` package are appropriate for a public carve-out. But v0.1.0 is over-scoped around segmentation, interactive tooling, analyses, and “fully generic” claims. The release should narrow its promise to a robust core path plus documented escape hatches, otherwise the first public users will hit installation, input-format, or algorithm-assumption failures before they ever evaluate the polarity method.

## Specific Issues

1. **§2/§3 Generality claim: “fully marker-agnostic” is too strong.**  
   Channel configurability does not make the algorithm biologically generic. Boundary-PCA assumes membrane-localized planar polarity signal, segmentable cell boundaries, and an interpretable axial orientation; migration-front analysis assumes collective migration geometry.  
   **Fix:** Reword to “channel-configurable for membrane-associated planar-polarity markers” and explicitly document unsupported cases: diffuse cytoplasmic markers, puncta-only markers, apical-basal polarity, sparse/non-confluent cells, non-front migration.

2. **§18 Quickstart acceptance is unrealistic with Cellpose-SAM and ND2.**  
   “First-time user with only Python + pip” is likely false because Cellpose, model weights, torch backend behavior, ND2 reading, and macOS/Linux binary dependencies are fragile.  
   **Fix:** Make the quickstart use pre-segmented masks or a small TIF demo by default; make raw ND2 + Cellpose-SAM a second-path “full pipeline” quickstart with environment checks.

3. **§5 CLI surface is too broad for a biologist-facing v0.1.0.**  
   `quantipy run` plus 11 subcommands makes the tool look like a framework, not a simple pipeline. Most users need run, inspect/report, validate, and maybe resume one failed stage.  
   **Fix:** Publicly document only `run`, `download-demo`, `validate`, and `debug`; hide or mark stage commands as “advanced/resume/debug” in CLI help and docs.

4. **§8 Interactive viewers are the most likely public-facing failure point.**  
   Matplotlib GUI behavior varies across backends, SSH, notebooks, macOS, and headless Linux; writing config from a GUI also risks silent reproducibility drift.  
   **Fix:** Keep `quantipy debug` read-only for v0.1.0, defer `front --interactive`, or make the tuner output a proposed YAML patch rather than mutating config. Prefer static QC overlays plus CLI parameter rerun before adding a richer GUI.

5. **§6 Single YAML is right, but the current schema hides mode-specific requirements.**  
   A minimal config with only `input.path` cannot reliably work across ND2, TIF, and mask inputs because channel semantics, pixel size, z-policy, FOV naming, and mask pairing are experiment-specific.  
   **Fix:** Keep one YAML, but define explicit profiles/modes: `raw_nd2`, `tif_stack`, `masks`. Provide `quantipy init-config --mode masks|tif|nd2` rather than pretending defaults are enough.

6. **§11 Lower-bound dependency pins are a release-risk multiplier.**  
   Scientific Python plus Cellpose plus image IO will break under future major versions. Lower bounds only are not kindness to users; they are deferred failure.  
   **Fix:** Use tested upper bounds or a constraints file for v0.1.0, plus `environment.yml` as the recommended reproducible install path. Relax later only after CI proves newer versions.

7. **§9 Validation story is credible but incomplete for referees.**  
   A QP-vs-Python parquet and scatter are useful, but without QP itself reviewers can only verify regeneration of the figure, not independent equivalence from raw inputs.  
   **Fix:** Be precise: “reproduces the shipped comparison analysis,” not “verify equivalence.” Ship provenance hashes, exact QP output columns used, source script lineage, and one frozen raw/reference mini-case if license/privacy allows.

8. **§15 Copy + refactor protects the research dir but creates divergence immediately.**  
   Once copied, bug fixes in the research pipeline and public package can fork silently, especially in the boundary-PCA and migration-front code.  
   **Fix:** After carve-out, designate the public package as upstream for reusable algorithms. The research repo should import or vendor a tagged public version, not keep evolving parallel core logic.

9. **§16 Module lift table under-specifies data contracts.**  
   It says where code comes from, but not what each module consumes/emits, which columns are stable, what coordinate conventions apply, or how axial angles are represented.  
   **Fix:** Add per-module contracts: input arrays, label conventions, angle units/ranges, per-cell schema, QC flags, FOV identifiers, pixel/micron handling, and migration-vector conventions.

10. **§4 `analyses/` is premature as a peer package.**  
   The listed analyses are mostly optoCelsr paper workflows, not core package behavior. Putting them beside `io`, `segment`, and `polarity` makes v0.1.0 look broader and more maintained than it is.  
   **Fix:** Move curated analyses to `examples/analyses/` or `quantipy_polarity/experimental/` until at least two external datasets justify stable APIs.

11. **§7 Output layout lacks failure/resume semantics.**  
   Numbered directories are good, but the spec does not define atomic writes, partial runs, overwrite behavior, or how a user resumes after segmentation succeeds and polarity fails.  
   **Fix:** Add `--resume`, `--force`, stage status metadata, and write-then-rename behavior for parquet/HTML outputs.

12. **§12 Demo data may create licensing/privacy/reproducibility exposure.**  
   A real FOV from an unpublished or lab-owned dataset in GitHub Releases needs explicit permission and metadata stripping.  
   **Fix:** Decide ownership and redistribution rights before implementation; otherwise ship synthetic plus a derived/de-identified TIF/mask bundle.

13. **§10 Claude-Code integration is over-prominent for a public scientific repo.**  
   It is useful for the author, but reviewers and biologists may read it as tool-specific clutter or as implying Claude is required.  
   **Fix:** Keep the files, but make README independent and put Claude integration under a short “Developer workflow” section, not a headline feature.

14. **§13 CI assumptions are optimistic.**  
   “CPU cellpose in seconds” on synthetic data is not a safe CI premise across macOS and Ubuntu, especially with model downloads.  
   **Fix:** Separate algorithmic CI from segmentation smoke tests. Use precomputed masks for e2e CI; run Cellpose tests as optional/nightly or allow them to skip when weights/backend are unavailable.

## Things The Spec Does Well

The core package layout is broadly sound: concern-based modules, `src/` layout, config validation, reproducible output snapshots, and a one-shot CLI are all worth preserving. The output directory is understandable and publication-friendly. Shipping synthetic fixtures plus validation artifacts is the right instinct. Dropping the vendored QP binary is also correct, as long as the validation claim is narrowed.

## Recommended Revisions Before Writing-Plans

1. Narrow the v0.1.0 promise: core polarity quantification from TIF/masks, raw ND2 + Cellpose as supported but not the only quickstart path.

2. Replace “fully marker-agnostic” with an assumptions-and-validity section that names biological and image-processing limits.

3. Collapse public CLI docs around `run`, `download-demo`, `validate`, and `debug`; mark all stage commands advanced.

4. Defer or reduce the interactive front tuner; keep front detection automatic plus reproducible static QC overlays.

5. Add formal data contracts for module boundaries and output parquet schemas before refactoring.

6. Add tested dependency constraints and make conda/environment install the recommended scientific path.

7. Make validation provenance stronger: hashes, QP version, source columns, frozen figure generation, and careful wording about what users can verify.

8. Move `analyses/` out of the stable core namespace or label it experimental.

9. Define resume/overwrite/failure behavior for every stage.

10. Resolve demo-data redistribution rights before uploading any real lab asset.
